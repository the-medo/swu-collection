# Feature Discord Tournament Results Plan

## Summary

Add a backend-only Discord notification feature that posts to the configured `#tournament-results` channel when a tournament transitions to imported.

The first message should ping the Discord role named `Tournament Results`, say:

```txt
Tournament Name results are in!
```

and attach the tournament screenshots generated by the screenshotter service as Discord embed images.

For v1, this should use the Discord REST API directly from the Bun/Hono backend. It does not need a long-running Discord client, gateway connection, `discord.js`, slash command handler, or extra process. The backend only needs to send one message to one known channel after import-side effects complete.

## Current Repo Baseline

- The screenshotter feature is already present under `server/screenshotter`.
- Screenshotter output is persisted in the generic `screenshotter` database table:
  - `scope_key = tournament:{tournamentId}`
  - `target = bracket | meta-leaders-and-base-all | meta-leaders-and-base-top8 | winning-deck`
  - `url` contains the public R2 URL
  - `status = success | failed`
- Shared screenshotter target constants already exist in `types/Screenshotter.ts`.
- The tournament screenshotter can run after import through `runTournamentScreenshotterAfterImport`.
- `runTournamentScreenshotterAfterImport` is currently called from:
  - `server/lib/live-tournaments/tournamentImportQueue.ts`
  - `server/routes/tournaments/_id/import-melee/post.ts`
  - `server/routes/tournaments/_id/import-from-blob/post.ts`
- The manual screenshot endpoint already exists:
  - `POST /api/tournament/:id/screenshots`
- The generic tournament update route can toggle `imported` through:
  - `server/routes/tournaments/_id/put.ts`
- The frontend tournament detail page exposes a "Marked as imported" switch, but this manual toggle should not send Discord notifications.
- No Discord backend service currently exists. The only Discord references are static/community links.
- There is no current database table for outbound Discord notification audit/idempotency.

## Key Decisions

- Use a real Discord bot token with the REST API.
- Do not add `discord.js` or a Discord gateway process for v1.
- Keep bot token and channel/role IDs server-side only.
- Use Discord IDs in env vars, not names. The channel can be named `#tournament-results` and the role can be named `Tournament Results`, but the backend should only depend on IDs.
- Ping exactly one role with `<@&ROLE_ID>`.
- Set `allowed_mentions` so only `DISCORD_TOURNAMENT_RESULTS_ROLE_ID` can notify users.
- Use public R2 screenshot URLs as embed `image.url` values. Do not download and upload Discord file attachments in v1.
- Read screenshots from the screenshotter manifest/DB after screenshot generation finishes.
- Send the Discord message after the tournament import data, statistics, thumbnails, and screenshotter have completed or been attempted.
- Discord success must not determine tournament import success. Import should remain successful if Discord fails.
- Add idempotency so a tournament does not notify the channel multiple times because a cron retry or route retry runs the post-import path again.
- Do not send Discord notifications from the manual imported toggle. Notifications should come only from real import workflows.
- Add a manual admin retry/dry-run route for operational use.

## Discord Setup

Create a Discord application/bot in the Discord Developer Portal and invite it to the server.

Required bot permissions for v1:

```txt
Send Messages
Embed Links
Mention Everyone
```

`Mention Everyone` is only needed when the `Tournament Results` role is not globally mentionable. Discord role pings notify members only when the role is mentionable or the bot has the `MENTION_EVERYONE` permission.

The bot needs access to the target text channel. The backend does not need the `MESSAGE_CONTENT` privileged intent because it is only creating messages through REST.

Helpful official docs:

- `POST /channels/{channel.id}/messages`: https://docs.discord.com/developers/resources/channel#create-message
- `allowed_mentions`: https://docs.discord.com/developers/resources/message#allowed-mentions-object

## Configuration

Add backend env vars:

```env
DISCORD_TOURNAMENT_RESULTS_ENABLED=false
DISCORD_BOT_TOKEN=
DISCORD_TOURNAMENT_RESULTS_CHANNEL_ID=
DISCORD_TOURNAMENT_RESULTS_ROLE_ID=
DISCORD_API_BASE_URL=https://discord.com/api/v10
DISCORD_TOURNAMENT_RESULTS_APP_BASE_URL=
```

Config behavior:

- `DISCORD_TOURNAMENT_RESULTS_ENABLED` gates automatic post-import sending.
- `DISCORD_BOT_TOKEN`, `DISCORD_TOURNAMENT_RESULTS_CHANNEL_ID`, and `DISCORD_TOURNAMENT_RESULTS_ROLE_ID` are required when sending is enabled.
- `DISCORD_API_BASE_URL` should default to `https://discord.com/api/v10`.
- `DISCORD_TOURNAMENT_RESULTS_APP_BASE_URL` should be optional. If omitted, build tournament URLs from the existing public app URL env if available, likely `VITE_BETTER_AUTH_URL`.
- Never expose the bot token to frontend code.
- Use separate screenshotter envs for screenshot capture; Discord should not own screenshotter configuration.

## Message Payload

Recommended v1 content:

```txt
<@&ROLE_ID> Tournament Name results are in!
```

Recommended payload shape:

```ts
{
  content: `<@&${roleId}> ${safeTournamentName} results are in!`,
  embeds,
  allowed_mentions: {
    roles: [roleId],
  },
}
```

Important details:

- The visible role mention must be present in `content`.
- `allowed_mentions.roles` should contain only the configured role ID.
- Do not include `parse: ['roles']`; allow the exact role by ID instead.
- Escape or sanitize tournament names before putting them in `content` so tournament names cannot create surprise formatting or visible fake pings.
- Discord supports up to 10 embeds per message. V1 needs at most 4 screenshots.
- Use one embed per screenshot so each image renders cleanly.

Recommended embeds, in order:

1. `bracket`
2. `meta-leaders-and-base-all`
3. `meta-leaders-and-base-top8`
4. `winning-deck`

Suggested first embed:

```ts
{
  title: `${tournament.name} results are in!`,
  url: tournamentUrl,
  description: [
    `Players: ${tournament.attendance}`,
    `[Open tournament](${tournamentUrl})`,
  ].join('\n'),
  image: { url: bracketScreenshotUrl },
  footer: { text: `SWU Base tournament:${tournament.id}` },
}
```

Suggested follow-up embeds:

```ts
{
  title: 'Meta - all decks',
  url: tournamentUrl,
  image: { url: metaAllScreenshotUrl },
  footer: { text: `SWU Base tournament:${tournament.id}` },
}
```

The `footer` marker is intentionally small but useful. Discord normal messages do not provide a reliable hidden metadata field that can be queried later, so the database audit table should remain the primary idempotency mechanism. The footer gives humans and emergency tooling a visible marker to identify which tournament a Discord message belongs to.

If no screenshots are available:

- Automatic path should still create a notification only if product preference is "notify no matter what".
- Preferred v1 behavior: send a text-only fallback with the tournament link, log that screenshots were unavailable, and include that in the notification audit row.
- If that feels too noisy during early rollout, gate text-only fallback with `DISCORD_TOURNAMENT_RESULTS_ALLOW_TEXT_ONLY=true`.

## Database Plan

Add a generic notification audit table so the feature is idempotent and future Discord use cases have somewhere to record delivery.

Suggested table name:

```txt
discord_notification
```

Suggested columns:

```txt
id uuid primary key
notification_type varchar(120) not null
scope_type varchar(80) not null
scope_id text
scope_key text not null
discord_channel_id text not null
discord_message_id text
status varchar(20) not null default 'pending'
error text
payload jsonb
sent_at timestamp/string
created_at timestamp/string not null
updated_at timestamp/string not null
```

For this feature:

```txt
notification_type = 'tournament-results'
scope_type = 'tournament'
scope_id = tournament id
scope_key = tournament:{tournamentId}
```

Suggested indexes/constraints:

- unique `(notification_type, scope_key)` for automatic idempotency
- unique `(notification_type, scope_type, scope_id)` if preferred by the local schema style; this is equivalent for tournament notifications because `scope_key = tournament:{tournamentId}`
- index on `(scope_type, scope_id)`
- index on `status`
- index on `sent_at`

Persistence behavior:

- Before an automatic send, check for an existing successful row for `(tournament-results, tournament:{id})`.
- If a successful row exists, skip sending.
- Also rely on the unique database constraint so concurrent import/retry paths cannot create two automatic sends for the same tournament.
- Insert or update a row with `status='pending'` or `status='sending'` before calling Discord.
- On Discord success, store `status='success'`, `discord_message_id`, `payload`, and `sent_at`.
- On Discord failure, store `status='failed'` and `error`.
- Manual `force` should be able to send again and update the row to the newest message ID.

Do not use Discord channel history as the primary duplicate check:

- Discord does not support a first-class hidden `tournamentId` metadata field on normal channel messages.
- Fetching channel history and scanning message content/embeds is limited, rate-sensitive, and can miss older messages.
- The reliable check is local state: `discord_notification` plus a unique tournament notification constraint.
- Optional recovery tooling can scan recent channel messages for the embed footer marker `SWU Base tournament:{tournamentId}` if the local DB is missing a row.

## Backend Structure

Add a small service under `server/lib/discord`.

Recommended files:

```txt
server/lib/discord/
  index.ts
  config.ts
  types.ts
  client.ts
  notificationLog.ts
  tournamentResults.ts
  runAfterImport.ts
  runTournamentResultsNotification.ts
```

### `config.ts`

Read and validate Discord runtime config:

```ts
getDiscordConfig(): DiscordConfig
getTournamentResultsDiscordConfig(): TournamentResultsDiscordConfig
```

Responsibilities:

- read env vars
- provide API base URL default
- validate required values only when sending is enabled or explicitly requested
- avoid throwing when feature is disabled and the caller is just checking whether to skip

### `client.ts`

Own the raw Discord REST call:

```ts
sendDiscordChannelMessage({
  channelId,
  botToken,
  payload,
}): Promise<DiscordMessageResponse>
```

Responsibilities:

- `fetch(`${apiBaseUrl}/channels/${channelId}/messages`, ...)`
- `Authorization: Bot ${botToken}`
- JSON request body
- parse success response
- include useful error body text in thrown errors
- handle `429` rate limits conservatively if Discord returns one

No extra dependency is needed.

### `tournamentResults.ts`

Own tournament-specific message preparation and sending:

```ts
sendTournamentResultsDiscordMessage({
  tournamentId,
  force,
  dryRun,
}): Promise<TournamentResultsDiscordResult>
```

Responsibilities:

1. Load the tournament.
2. Validate that it exists and is imported.
3. Build the public tournament URL.
4. Load screenshot rows from `screenshotter` for `scope_key = tournament:{id}`.
5. Keep only useful successful screenshots, ordered by known target order.
6. Build Discord message content and embeds.
7. Respect idempotency through `discord_notification`.
8. Send the message unless `dryRun` is true.
9. Persist success/failure status.

### `notificationLog.ts`

Own database behavior around `discord_notification`.

Suggested helpers:

```ts
getNotificationByScope(notificationType, scopeKey)
markNotificationSending(input)
markNotificationSuccess(input)
markNotificationFailed(input)
```

Keep DB idempotency separate from the Discord fetch client so the REST client remains testable.

### `runAfterImport.ts`

Add a shared wrapper for automatic post-import usage:

```ts
runTournamentResultsDiscordAfterImport(tournamentId: string)
```

Behavior:

- return `skipped` when `DISCORD_TOURNAMENT_RESULTS_ENABLED` is not enabled
- return `skipped` when a successful notification already exists
- call `sendTournamentResultsDiscordMessage`
- catch errors and return a structured result instead of throwing into the import flow
- log failures for Sentry visibility

### `runTournamentResultsNotification.ts`

Add a script for manual operations:

```txt
bun server/lib/discord/runTournamentResultsNotification.ts --tournament-id {uuid}
```

Optional flags:

```txt
--force
--dry-run
--run-screenshotter
--json
```

Behavior:

- `--dry-run` prints the message payload without sending.
- `--force` bypasses the existing-success skip.
- `--run-screenshotter` runs `captureScreenshots` before building the Discord payload.
- Exit `0` on send/dry-run success, `1` on Discord failure, `2` on config/validation errors.

## Screenshot Selection

Use the screenshotter DB table as the app-facing source of screenshot URLs.

Query:

```txt
scope_key = tournament:{tournamentId}
status = success
target IN (
  bracket,
  meta-leaders-and-base-all,
  meta-leaders-and-base-top8,
  winning-deck
)
```

Order:

```txt
bracket
meta-leaders-and-base-all
meta-leaders-and-base-top8
winning-deck
```

Notes:

- Do not scrape R2 directly.
- Do not assume the manifest URL is enough; the DB table is easier to query and already holds current public URLs.
- Ignore failed rows for embeds.
- If a failed screenshot row still has an old preserved URL, do not use it in v1. That avoids quietly posting stale images.
- Cap embeds at 10 even if future screenshot targets are added.

## Import Integration Plan

The safest implementation is a shared post-import side-effect function that runs screenshotter first and Discord second.

Recommended helper:

```ts
runTournamentImportedSideEffects(tournamentId: string)
```

Responsibilities:

1. Run screenshotter after import if `SCREENSHOTTER_RUN_AFTER_IMPORT` is enabled.
2. Run Discord notification after screenshotter finishes or skips.
3. Catch/log side-effect failures so import success is not rolled back.
4. Return structured results for logs and scripts.

Wire the helper into these paths:

### Import queue

File:

```txt
server/lib/live-tournaments/tournamentImportQueue.ts
```

Current flow already calls `runTournamentScreenshotterAfterImport`. Replace or wrap that call so Discord happens immediately after screenshotter.

The queue should notify only after the tournament was newly imported. The existing "already imported" skip should not send Discord.

### Melee import route

File:

```txt
server/routes/tournaments/_id/import-melee/post.ts
```

When `markAsImported` is true:

1. update `tournament.imported = true`
2. compute stats
3. update tournament group stats
4. generate thumbnails
5. run post-import side effects

Because this route runs work in a background promise, keep Discord inside that same background chain.

### Import from blob route

File:

```txt
server/routes/tournaments/_id/import-from-blob/post.ts
```

When `markAsImported` is true, replace the current fire-and-forget screenshotter call with the shared post-import side-effect helper.

### Manual imported toggle

File:

```txt
server/routes/tournaments/_id/put.ts
```

Do not wire Discord notifications into this path.

The manual "Marked as imported" switch should remain a local data-maintenance action. It may continue to update tournament state, recompute stats, or trigger existing maintenance behavior, but it should not post to Discord.

The idempotency table still protects the real import workflows if two automatic paths race.

## API Plan

Add an admin-only manual endpoint:

```txt
POST /api/tournament/:id/discord-results
```

Request shape:

```ts
{
  force?: boolean;
  dryRun?: boolean;
  runScreenshotter?: boolean;
}
```

Response shape:

```ts
{
  message: string;
  data: {
    status: 'sent' | 'skipped' | 'dry-run' | 'failed';
    tournamentId: string;
    discordMessageId?: string;
    payload?: unknown;
    reason?: string;
    error?: string;
  }
}
```

Authorization:

- Require logged-in user.
- Require admin access using `requireAdmin`, matching the screenshotter endpoint.

Route wiring:

- Create `server/routes/tournaments/_id/discord-results/post.ts`.
- Add it to `server/routes/tournament.ts`.

No special route timeout should be needed unless `runScreenshotter` is true. If `runScreenshotter` is supported through this route, reuse the existing screenshotter timeout style and set `app.use('/api/tournament/:id/discord-results', timeout(300000))`.

## Types

Keep Discord-specific types server-local unless a frontend admin view needs them.

Suggested server types:

```ts
type DiscordConfig = {
  apiBaseUrl: string;
  botToken?: string;
};

type TournamentResultsDiscordConfig = DiscordConfig & {
  enabled: boolean;
  channelId?: string;
  roleId?: string;
  appBaseUrl: string;
};

type TournamentResultsDiscordResult =
  | { status: 'skipped'; tournamentId: string; reason: string }
  | { status: 'sent'; tournamentId: string; discordMessageId: string }
  | { status: 'dry-run'; tournamentId: string; payload: DiscordCreateMessagePayload }
  | { status: 'failed'; tournamentId: string; error: string };
```

If the manual endpoint will be consumed by frontend code, add shared response types under `types/DiscordNotification.ts`.

## Failure Handling

Expected failure cases:

- Discord feature disabled
- missing bot token/channel ID/role ID
- tournament does not exist
- tournament is not imported
- no successful screenshots exist yet
- Discord channel cannot be posted to
- role ping does not notify because role is not mentionable and bot lacks permission
- Discord returns a rate limit
- Discord returns a validation error for payload/embed limits
- DB insert/update for notification audit fails

Recommended behavior:

- Missing config during automatic import path returns `skipped` or `failed` from the wrapper but does not throw into import.
- Validation errors from the manual endpoint return `400`.
- Discord send errors return `500` from the manual endpoint.
- Automatic path logs send failures and records `status='failed'`.
- Duplicate automatic notifications return `skipped`.
- If screenshots fail, still include the tournament link. Whether to send text-only should be controlled by product preference/config.

2. Run:


   - role mention in `content`

   - public tournament URL
   - embed footer marker `SWU Base tournament:{tournamentId}`

1. Configure a test Discord channel and role.
2. Enable env vars locally or in staging.
3. Run:

```txt
bun server/lib/discord/runTournamentResultsNotification.ts --tournament-id {uuid}
```

## Settled Decisions

- V1 uses Discord REST directly from Bun/Hono.
- V1 uses a bot token, not a webhook, so the setup can grow into slash commands and interactive bot features later.
- V1 posts to a known channel ID for `#tournament-results`.
- V1 pings a known role ID for `Tournament Results`.
- V1 uses screenshotter R2 URLs as embed images.
- V1 does not add a Discord gateway process.
- V1 records sent notifications for idempotency and debugging.
- V1 includes a small `SWU Base tournament:{tournamentId}` footer marker in Discord embeds, but local DB idempotency is the reliable duplicate prevention mechanism.
- V1 does not notify from the manual "Marked as imported" toggle.
- Tournament import success does not depend on Discord send success.
